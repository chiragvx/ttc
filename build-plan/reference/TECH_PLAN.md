# Final Technology Plan: Grounded Two-Plane Co-Modeling Platform

Spine: **Grounded Monolith** (Plan 1) — lowest integration risk, fastest credible MVP, and the cleanest articulation of the three non-negotiable inversions (LLM emits validated deltas not code; deterministic Jinja+build123d templating; grounded solvers off the hot path). Grafted onto it: **Anvil's** Manifold-preview/OCCT-authoritative kernel split + real OCAF/TNaming persistent identity (the single best geometry idea on the table, and the fix for build123d's fatal naming gap), Anvil's segmented-assembly cohesive-joint FEA, and **CloudForge's** Postgres Row-Level-Security multi-tenancy, Temporal-durable solver DAGs, three-tier wire protocol, and its correct merge stance (CRDT on scalar sliders, human-reviewed diff on geometry).

---

## 1. Recommended architecture (one paragraph)

A **single Python/FastAPI monolith** that physically separates two clocks into two planes. The **Interactive Plane** lives in-process and does only cheap, closed-form, deterministic work — schema-validate a slider delta against a hardened ledger, clamp to precomputed bounds, and answer the 30 Hz HUD from analytic proxies (mass = Σρᵢ·Vᵢ, CG = mass-moment/total over a positioned BOM, print-time = extrusion-volume/flow-rate); the browser interpolates a precomputed morph-target mesh family on the GPU so geometry *appears* to move during drag. The **Truth Plane** is everything slow and grounded — OCCT B-rep regeneration, BRepCheck/ShapeFix validity repair, tessellation, Gmsh+CalculiX FEA, AVL/XFOIL aero, p-k flutter, PrusaSlicer slicing, NSGA-II/qEHVI optimization — run as **durable async jobs**, content-addressed by a pinned-toolchain fingerprint, never blocking a frame and never re-invoked during replay. Every generated script is executed inside a **pre-warmed Firecracker microVM** with egress-deny, RLIMIT, and a host-side wall-clock SIGKILL, because OCCT's C++ `Standard_Failure`/access violations segfault CPython and degenerate booleans spin uninterruptibly. **The LLM never originates a safety scalar and never emits free Python** — it emits Pydantic-validated parameter deltas, a deterministic Jinja2+build123d templater renders the script, and real solvers produce every FS/stall/flutter/range number; where inputs are missing the system shows **"unknown" (blocking export)**, never a fabricated green light.

**Why this layer was deliberately omitted from the PRDs:** the entire missing substrate — sandbox, solver farm, surrogate training, persistent topological naming, the tiered clock, the determinism fingerprint — is precisely what makes the headline promises (150 ms full recompile, an LLM "Structural Validator" emitting FS, a "deterministic replay" of non-deterministic producers) *impossible as stated*. Naming it forces the PRDs to admit that the single clock is a fiction, that the LLM cannot live in the safety loop, and that "single source of truth" is currently decorative. The omission is load-bearing: the PRDs sell a demo-shaped fantasy, and this layer is the unglamorous engineering that converts it into an honest, certifiable tool.

## 3. The hot path: a tiered latency model

The PRD's single clock is physically false (one OCCT boolean/loft/fillet = 100 ms–seconds; tessellating a 2.8 m skin to 0.2 mm = 10⁵–10⁶ triangles at tens–hundreds of ms; a real 42 h-part slice = tens of seconds; one FEA solve = seconds–minutes; an NSGA-II front = 10³–10⁵ evals; OCCT has **no** incremental B-rep edit). So the targets are split across three honest tiers:

**Tier 0 — Interactive (<16–33 ms, 30 Hz, in-process, zero server kernel work).**
During slider *drag*: the client GPU-interpolates a precomputed discrete mesh family across the bounded slider range (zero round-trips/frame), and the HUD ticks from closed-form arithmetic in the FastAPI process — mass = Σρᵢ·Vᵢ over precomputed per-parameter volume sensitivities + positioned BOM, CG = mass-moment/total in one declared body datum frame, print-time = extrusion-volume/flow-rate. Each number is <1 ms. **30 Hz HUD: MET.** No LLM, no OCCT, no FEA, no slicer, no broker on this path.

**Tier 1 — Kernel regen (~1–5 s, on slider *release*, async via warm microVM).**
The "150 ms" target is **honestly relaxed and re-scoped**: 150 ms applies to *what a human perceives as the slider responding* (Tier 0 morph + analytic HUD), which is MET. The actual B-rep regeneration is debounced to release: a warm Firecracker worker renders the Jinja+build123d template, runs OCCT (or Manifold for isolated local edits), BRepCheck/ShapeFix, the **post-compile round-trip verifier** (query OCC bbox/topology, assert every ledger dimension within tolerance so a HARD_LOCK 4.5 mm pin cannot silently drift to 4.50001 mm), tessellates, and streams a glTF delta. For a genuinely isolated local feature branch this can approach ~150 ms; for multi-feature/global edits it is 1–5 s and is shown as a proxy preview until it settles. The PRD's own 20:45:00→20:45:02 (2 s) example is correctly reclassified as **this tier, not the hot path.**

**Tier 2 — Grounded solvers & optimization (seconds–minutes, Temporal DAG, off-interactive).**
Gmsh+CalculiX FS/buckling/modal, AVL/XFOIL, p-k flutter, PrusaSlicer slice, and the NSGA-II/qEHVI MOO. The **15 s Pareto: MET only as a real surrogate-evaluated MOO** over an explicit objective vector `[mass, min-FS, buckling-load-factor, flutter-margin, print-time, L/D]`, evaluated through GP/POD surrogates (µs/eval) with **constraints embedded via constraint-domination** (buckling+modal+flutter feasibility *inside* the optimizer, not as post-hoc gates), multi-start, and hypervolume/KKT-feasibility convergence — **not** the noise-floor ΔW<0.1 g greedy descent (which is ~2e-5 of 4.65 kg, certifies neither optimality nor feasibility, and false-converges when thinning and rib-adding cancel). The returned front is then re-verified by real CalculiX/flutter runs asynchronously before export. **Export gates (FS≥1.5, stall, range≥100 km, flutter, CG%MAC): MET truthfully** — every number is a grounded solver output with error bars; missing inputs block with "unknown."

## 4. Event-sourcing / branching / merge — concretely

**Storage.** Append-only `events` table in Postgres, one ordered stream per project, optimistic concurrency on a per-project monotonic `sequence`. Events are split into two classes:
- **FACTS** (replayable user intent): parameter mutations, locks, requirement edits, NL intents, accepted AI-deltas, branch/merge ops.
- **DERIVATIONS** (recomputed outputs, persisted as *first-class* events): the exact rendered build123d script, resulting STEP/B-rep + mesh hashes, surrogate + solver verdicts, telemetry snapshots — each content-addressed in MinIO and stamped with a **toolchain fingerprint** (OCCT/build123d/Python wheel hashes, BLAS thread count, rounding/FMA flags) plus **content-hashes of every external dataset consumed** (material allowables, thread tables, airfoil .dat).

**Replay / time-travel.** A **pure fold over stored FACTS, rehydrating DERIVATIONS by hash.** It **never re-invokes the LLM, optimizer, or any solver** — that is the canonical fix for the event-sourcing anti-pattern of replaying non-deterministic producers. v2.4.1 reconstructs byte-identically next quarter because the rendered script + B-rep + mesh are stored artifacts under a pinned fingerprint, satisfying AS9100 configuration control. If the current toolchain fingerprint differs from the stored one, replay serves the stored artifact and flags a **regeneration-mismatch** rather than silently recomputing.

**Branching.** A branch is a new stream forked at a `sequence` point sharing all prior FACTS by reference (copy-on-write); branch comparison diffs FACTS + the latest DERIVATION verdicts (mass, FS, L/D, range) side by side, and additionally **which requirements each branch satisfies** (a requirements/verification matrix is a first-class ledger artifact, so "which branch is better" is answerable, not just parameter-by-parameter).

**Conflict strategy — split by safety criticality (the correct call grafted from CloudForge, grounded):**
- **Scalar bounded-slider parameters** converge via a **Yjs/Automerge CRDT** for live multi-user co-editing — but bound to **invariant-aware merge resolvers**: a merge that would violate `bounds[0]≤value≤bounds[1]`, a HARD_LOCK, or a coupled invariant (skin_thickness≥min-wall) is rejected to a conflict state, because CRDTs guarantee structural convergence but **not** semantic/safety-invariant preservation.
- **Geometry / topology merges** (PRD3 §3.3's "programmatically alter mounting holes") are **never auto-applied.** They land as an **AI-PROPOSED diff** with granular per-change accept/reject and an explicit **engineer sign-off state** before committing to the ledger — distinguishing "AI-proposed" from "engineer-reviewed," which the requirements-traceability and AS9100 audit depend on. Last-writer-wins on flight geometry is forbidden.

## 5. The single biggest technical bet — and its fallback

**The bet: persistent topological identity via baked-in feature tags + OCAF/TNaming on top of build123d/OCP.** This is the load-bearing keystone — it is the *single root cause* recurring across kernel mutation, picking, anchored-HUD, delta-mesh diffing, optimizer feature-targeting, and merge. If feature bindings survive regeneration and sub-shape renumbering, the whole platform (sliders that target the right wall, HUDs that stay anchored, an optimizer that thins *the* low-stress rib, a merge that re-binds *the* mounting hole) works. If they don't, every one of those silently re-binds to the wrong geometry and the "single source of truth" is a lie. The risk: **OCP's pybind11 surface for OCAF/TNaming documents is incomplete**, so the clean C++ path Anvil gets natively must be approximated from Python with generator-emitted string tags + a geometric sidecar map — fragile across booleans that split/merge faces.

**The fallback (staged, in priority order):**
1. **Generator-deterministic tagging first:** because we *invert codegen* (the templater, not the LLM, authors every feature), we control creation order and can emit stable, human-meaningful tags at build time (`wing.rib[3].web`, `joint[7].pin_bore`) — most identity needs are satisfiable without TNaming's full history-tracking, since the template *knows* what it made.
2. **Geometric re-identification as backstop:** match faces/edges across regen by invariant signatures (centroid, area, normal, adjacency) within tolerance, flagging ambiguous matches for human confirmation rather than silently re-binding.
3. **If both prove too fragile for production:** execute Anvil's graft in full — drop to **raw OCCT 7.8 in C++ behind a thin Rust/cxx or pybind worker** for the authoritative regen path *only*, where OCAF+TNaming genuinely exist, keeping the rest of the Python monolith intact. This is a contained, single-service escalation (the regen worker), not a rewrite — the two-plane architecture, event store, sandbox, and frontend are unchanged.

This staging lets the MVP ship on the cheap Python path and buys the option to harden the one component that most needs it, without betting the timeline on the rarest hire (Rust + OpenCASCADE C++ FFI) up front.
